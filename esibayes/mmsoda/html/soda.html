Description

soda --addtools

If the MMSODA software is run offline (as opposed to being run compiled on the cluster), this command adds the ./model, ../mmsoda/comms, ../mmsoda/enkf, ../mmsoda/mo, ../mmsoda/mmlib, ../mmsoda/tools, ../mmsoda/visualization directories to the MATLAB search path.

soda --addtools --quiet

Same as soda --addtools, but without any verbose feedback.

soda --builddb

Builds the database that is used by the docsearch command. See also builddocsearchdb. Only used offline.

soda --docinstall

Installs the documentation by writing the 'soda-styles.css' style sheet file using the user-specific styles; adds the necessary directories to the MATLAB search path by calling soda --addtools --quiet; builds the the docsearch documentation by calling soda --builddb. Only used offline.

soda --docinstall --quiet

Same as soda --docinstall, but without any verbose feedback.

soda --help

Prints a help message to the screen.

soda --showoptions

Prints a list of all authorized options for the soda configuration. See also table below.

soda --rmpath

Removes the MMSODA directories from the MATLAB search path.

soda --root

Prints the root directory of the MMSODA Toolbox for MATLAB.

Calls the SODA parameter optimization algorithm. soda() expects 3 directories to be present in the current working directory, some of which need to contain particular files. Furthermore, the current working directory should contain a copy of `Makefile' to be able to compile the MATLAB code into an executable, which can then be run on the cluster. It further requires an executable script 'run-mmsoda.sh' or a PBS jobscript 'jobscript-mmsoda.pbs'.

So the general structure of files and folders is as follows:

  1. mmsoda (directory)
  2. <current working directory> (directory)
    1. data (directory)
      • constants.mat (binary file containing the model constants)
        •
    2. model (directory)
      • <user's model m-files>
      • <user's objective function m-files>
        •
    3. results (directory)
      • conf.mat (binary file containing soda's configuration variables)
      • X-Y-tempstate.mat (file)
      • X-results-enkf-evals-Z-Z.mat (file)
      • X-results.mat (file)
        •
    4. Makefile (file)
    5. run-mmsoda.sh (file) and/or jobscript-mmsoda.pbs (file)
    4.

Note that in the above list, X can be one of four strings 'bypass', 'scemua', 'reset', or 'soda', depending on the type of optimization that is being performed; Y is a string 'so' or 'mo', depending on whether the optimization is single-objective or multi-objective; Z is a number indicating which parameter combinations are contained in the file.

Note that the 'mmsoda' folder must be located at the same level as the current working directory.

Further note that 'X-Y-tempstate.mat', 'X-results-enkf-evals-Z-Z.mat', and 'X-results.mat' are output files and are therefore generated by the SODA software (as opposed to by the user). In contrast, 'constants.mat' and 'conf.mat' must be created by the user prior to starting the optimization.

The 'data', 'model', and 'results' directories must be located in the directory that the soda() function is called from (which usually is the current working directory). All three folders must be present, even though your specific setup may not require all three to have anything in them. You must adhere to the naming convention as displayed above, at least with regard to the folder names and file names that are displayed in bold.

Further down you'll find an overview of the meaning of soda()'s output variables. In the meantime, the table below lists what configuration variables can be included in conf.mat and what their intended purpose is. Individual configuration variables may or may not be required, depending on what type of optimization you are doing:

  • R: always required configuration variable.
  • R*: sometimes required configuration variable.
Furthermore, some configuration variables are not intended to be user-settable, while others are:
  • U: can be set by the user;
  • N: should not be set by the user;
Configuration variables that are not assigned by the user will be assigned their default values, either from the file 'mmsoda/soda-default-settings.ini' or in the file as specified by the useIniFile configuration variable.

assimilate N N U U Logical row vector with the same number of elements as conf.priorTimes. True values indicate at which times data assimilation should occur.
avgWalltimePerParSet N N N N This field is used to record how long it took on average to evaluate one parameter combination.
badScore U U U U This value is assigned as objective score when an error occurs during evaluation of the model. This opion is useful when certain parts of the parameter space result in model crashes, for instance due to integration errors.
convMaxDiff U U U U This scalar value is used in determining whether the optimization has converged. Is a more stringent criterion than just assuming that convergence has been achieved when the Gelman-Rubin values in the critGelRub array are less than 1.2. Its default value is 0.001.
convUseLastFraction U U U U This scalar real valued option determines what fraction of the record of samples is used in determining whether the optimization has converged.
converged N N N N Logical scalar indicating which whether the optimization has converged acording to the Gelman-Rubin criterion.
covModelPert N N N U,R Covariance of the model perturbations (white noise).
covObsPert N N N U,R Covariance of the observation perturbations (white noise).
critGelRubConvd U U U U The Gelman-Rubin criterion must be less that conf.critGelRubConvd for convergence to be achieved. Its default value is 1.2. See also convMaxDiff.
drawInterval U U U U This scalar integer value determines how often the call to the visualization script is made. It is measured in generations, not model evaluations.
doPlot U U U U

This logical scalar detemines whether any visualization is performed. Note that doPlot can be true even when running compiled on a remote cluster, but that this only works when X forwarding has been enabled for your SSH connection to the cluster and that it requires an X environment to run on your local machine. Since Windows does not use X for its graphics, Windows users need an additional program (for instance Xming) to be able to view any MATLAB figures over SSH.

See also drawInterval.

initValuesKF N N U,R U,R When conf.initMethodKF is set to 'reference', initValuesKF contains the initial values of the state variables that are subject to state updating by the Ensemble Kalman Filter.
initMethodNOKF N N U,R U,R This string option determines what method is used to initialize the variables that are not subject to Ensemble Kalman Filter updating, but that are unique to individual ensemble members. Currently, only 'reference' is implemented.
initValuesNOKF N N U,R U,R When conf.initMethodNOKF is set to 'reference', initValuesNOKF contains the initial values of the variables that are not subject to state updating by the Ensemble Kalman Filter, but that are unique to individual ensemble members.
isSingleObjective N N N N Logical scalar indicating whether the optimization is single-objective or not. Note that its value is always the opposite of conf.isMultiObjective.
isMultiObjective N N N N Logical scalar indicating whether the optimization is multi-objective or not. Note that its value is always the opposite of conf.isSingleObjective.
jumpRate U U U U This scalar real value acts as a scaler on the covariance matrix that is used to generate new samples in sodaGenerateOffspring. Its default value is 2.4*sqrt(conf.nOptPars).
llCols N N N N For single-objective optimization, llCols is a scalar integer that indicates which column in evalResults, sequences, and metropolisRejects holds the log likelihood information. For multi-objective optimization, llCols is a row vector of integers, that indicates which columns in evalResults, sequences, and metropolisRejects hold the objective scores (which need not necessarily be all log likelihood values).
modeGelRub U U U U

This string option determines how the Gelman-Rubin criterion is calculated. If set to 'strict', the SODA algorithm will calculate the second term from the equation from the paper as is, i.e.

2*(((m+1)*(n-1))/(m*n^2))

If set to 'loose', the SODA algorithm will use a slightly different version, viz.

2*(((m+1)*(n-1))/(m*n)^2),

the difference being in the placement of the parentheses in the denominator. Default is 'loose'.
modeStr U,R U,R U,R U,R This string option can be one of the following: 'bypass', 'scemua', 'reset', or 'soda'. It determines what kind of optimization will be performed.

If modeStr is set to 'bypass', than the call to the model function file is bypassed, hence the name. This is useful if the objective function calculates the log-likelihood directly, for instance on known probability distributions (benchmark functions).

If modeStr is set to 'scemua', the SODA algorithm does not do any state updating, and thus behaves like SCEM-UA (hence the name).

If modeStr is set to 'reset', the SODA algorithm does perform state updating, but with full confidence on the measurements. In this mode the ensemble consists of just one member, and is therefore cheaper in terms of computation time.

If modeStr is set to 'soda', the SODA algorithm does perform state updating according to an Ensemble Kalman Filter.

Depending on what mode is selected, the algorithm will remove fields from conf or reset some of its fields. Also, the required fields are not the same for each mode; for instance, in 'bypass' mode, there is no need for the modelName field.
modelName N U,R U,R U,R Name of the model whose parameters are calibrated. Note that the model should be located in a subfolder ./model.
namesNOKF N N U,R* U,R* Cell array of strings, listing the names of the variables that are not subject to state updating, but that are unique to individual members in the ensemble. (R*: required if there are variables with values that are unique for individual ensemble members).

nGelRub U U U U Minimum number of generations for convergence to be achieved.

Output argument Description
evalResults

This 2-D array is ordered as follows:

For single-objective optimization, column conf.evalCol holds the model evaluation number; columns conf.parCols hold the parameter combination; column conf.llCols holds the log-likelihood score.

For multi-objective optimization, column conf.evalCol holds the model evaluation number; columns conf.parCols hold the parameter combination; columns conf.llCols hold the log-likelihood scores (or other types of scores) associated with each objective function. Column conf.paretoCol hold the Pareto score.
critGelRub This array holds the Gelman-Rubin convergence information. The scale reduction criterion is calculated after every generation. Column 1 in critGelRub holds the iteration number of the last model evaluation before the Gelman-Rubin scale reduction criterion was assessed; columns 1+1:N hold the value of the Gelman-Rubin scale reduction criterion for each parameter, with N the number of calibration parameters.
sequences Rows and columns of this array hold the exact same information as evalResults, but divided over conf.nCompl Markov chains.
metropolisRejects This array has the same size and dimensions as the sequences array, but instead it holds the parameter space samples that were rejected by the Metropolis scheme.
conf This structure array holds the configuration variables as specified by the user, with some fields added by the SODA algorithm.
